Scaling & Systems Design Document

1. Production Architecture (Scaling to 200,000+ Players)

To scale from 2,000 static players to a highly dynamic, concurrent player base, we decouple match score submission from live query evaluation. This prevents high ingestion rates from locking database tables or slowing down reads.

Stateless API Layer

The FastAPI backend runs as containerized pods inside an auto-scaling cluster (e.g., Kubernetes). A load balancer distributes incoming HTTP requests. API servers validate payload structures, generate unique request IDs, and immediately drop writes onto a messaging bus rather than writing directly to a transactional database.

Asynchronous Ingestion (Kafka / AWS SQS)

A message queue acts as a buffer. Under high load (e.g., millions of submissions during weekends), it absorbs ingestion spikes. Asynchronous workers consume batches from Kafka and stream them into the primary database, ensuring the APIs remain highly responsive (latencies < 20ms).

Data Layer Partitioning & Caching
  • Primary Store (PostgreSQL): Stores raw telemetry, historical aggregation tables, and user profiles. PostgreSQL tables are horizontally partitioned by region and creation_date, preventing single large tables from causing slow lookups.
  • Hot Leaderboard Cache (Redis Sorted Sets): Generating full leaderboards on PostgreSQL using aggregate functions is expensive. Instead, Redis ZADD is used to update players' score ranks in $O(\log N)$ time. Reads (ZRANGE) retrieve rankings instantaneously (< 5ms).

2. Anti-Cheat & Security Policy

We implement a layered security check. In production, Rule Checks are performed by stream-processing engines (e.g. Apache Flink) running directly on Kafka topics. When a suspicious event fires, it posts an alert. Player Z-scores are computed by background batch engines (e.g. Apache Spark) operating on the primary database, running every few minutes to flag slow-building anomalies.

3. Monitoring Strategy

  • Prometheus: Scrapes application-level metrics (HTTP response distributions, active connections, queue processing lag, worker status).
  • Grafana: Visualizes Prometheus metrics to display real-time dashboard tracking:
    • API Latency: p95 and p99 server response latencies.
    • Queue Lag: Number of unconsumed messages in SQS/Kafka.
    • Security Telemetry: Count of flagged cheaters per minute.
